# 原生 Android 客户端建设提案

## 1. 背景

当前项目已经具备：
- Go 后端：负责认证、HTTP 代理、WebSocket 代理、媒体上传、抖音扩展、mtPhoto 接入、视频抽帧等能力
- Vue 网页端：已经覆盖登录、身份管理、会话列表、聊天收发、媒体管理、抖音下载、mtPhoto、视频抽帧等完整交互

用户当前希望：
- 在**不替换现有 Go 服务端**的前提下，新增一个**原生 Android 客户端**
- 要求 Android 端在业务能力上与网页端保持一致，能够直接复用当前 HTTP / WebSocket / 上传链路

这不是简单的页面改写，而是一次新的客户端产品建设：
- 需要重新设计移动端原生架构
- 需要适配复杂且不完全统一的后端接口
- 需要覆盖实时消息、媒体上传/预览、长列表、多弹层业务与扩展工具模块

---

## 2. 目标

### 2.1 核心目标
- 新增一个使用 **Kotlin + Jetpack Compose** 的原生 Android 客户端
- 保持与当前 Go 服务端兼容，不要求后端为 Android 单独重写接口
- 业务功能覆盖网页端已上线能力
- 为后续 Android 持续迭代建立清晰、可维护的分层架构

### 2.2 交付目标
- 提供 Android 项目骨架与模块化工程结构
- 提供 HTTP / WebSocket / 上传 / 下载 / 缓存能力封装
- 提供与网页端对齐的页面与业务模块
- 提供本地状态管理、离线缓存、错误处理、调试日志与能力降级方案

---

## 3. 范围

### 3.1 范围内
本次 Android 客户端需要覆盖以下网页端能力：

1. 认证与身份
   - 访问码登录
   - JWT 校验与持久化
   - 本地身份列表/创建/更新/删除/选择

2. 聊天主流程
   - 会话列表（历史 / 收藏）
   - 聊天页消息展示
   - WebSocket 连接、断线重连、sign 绑定
   - 文本消息、表情消息、匹配/中断、拉黑、清空记录等操作
   - 消息历史分页与本地消息合并

3. 媒体能力
   - 图片/视频上传
   - 历史媒体浏览
   - 全部上传媒体管理
   - 聊天历史图片/视频查看
   - 删除/批量删除/重新上传/查重

4. 扩展业务模块
   - 抖音下载与导入上传
   - 抖音收藏作者 / 作品 / 标签管理
   - mtPhoto 相册/目录浏览、缩略图/原图获取、导入上传
   - 视频抽帧任务创建、任务列表、详情、取消、继续、删除

5. 系统与配置
   - 图片服务地址配置
   - 系统配置读取与更新
   - WebSocket 连接统计、强退名单清理等系统页面能力

### 3.2 范围外
- 修改上游聊天服务协议
- 重写现有 Go 服务为移动端专用 API
- iOS 客户端
- 像素级复刻网页端 CSS 样式

## 4. 用户与场景分析

### 4.1 用户画像
- **高频聊天用户**：主要使用身份切换、匹配聊天、图片/视频发送
- **重度媒体用户**：频繁上传本地素材、复用历史图片、管理媒体库
- **内容搬运/整理用户**：依赖抖音解析、mtPhoto 导入、视频抽帧等扩展工具
- **维护者/高级用户**：需要查看系统状态、连接统计与服务配置

### 4.2 关键场景
1. 用户打开 App，输入访问码登录，选择身份并开始聊天
2. 用户在聊天过程中发送文本、表情、图片、视频，并查看历史图片
3. 用户在图片管理中整理已上传内容，避免重复上传
4. 用户从 mtPhoto 或抖音模块选择媒体导入并发送
5. 用户在抽帧中心对视频创建任务并查看结果
6. 用户在设置中调整图片服务与系统配置，并查看连接状态

### 4.3 痛点
- Web 端功能多且入口分散，原生端若没有清晰架构，维护成本会迅速上升
- 后端返回结构不统一，客户端很容易出现解析与异常处理混乱
- 聊天与媒体是高频核心链路，任何架构设计不当都会影响体验
- 抖音、mtPhoto、抽帧依赖外部能力，需要具备可降级展示能力

---

## 5. 方案对比

## 方案 A：Jetpack Compose 模块化架构（已选）

### 优点
- 适合复杂状态与多页面、多弹层交互
- UI 与状态管理更统一，便于实现聊天页、媒体页、任务中心等复杂界面
- 更适合后续持续增加功能与维护
- 与 Kotlin 协程、Flow、Navigation、Room、DataStore 组合成熟

### 缺点
- 初始搭建成本高于 XML
- 团队若对 Compose 不熟，前期学习成本更高

### 适用性结论
最适合当前“功能完整迁移 + 长期演进”的目标。

## 方案 B：XML + Fragment

### 优点
- 传统方案稳定，资料多
- 老项目迁移经验更容易复用

### 缺点
- 多状态页面、复杂弹层、列表与媒体预览实现成本更高
- 后期维护与跨模块统一性较弱

### 适用性结论
适合保守交付，但不适合当前大量复杂交互的长期维护。

## 方案 C：Compose + 严格分阶段交付

### 优点
- 风险最低，便于先交付聊天核心链路
- 适合资源有限时控制范围

### 缺点
- 会延后抖音、mtPhoto、抽帧等能力的一致性完成时间
- 与用户“所有功能一致”的预期存在阶段性落差

### 适用性结论
可作为实施节奏，不作为本次技术架构主方案。

---

## 6. 选型结论

本次采用：**方案 A：Jetpack Compose 模块化架构**。

选型理由：
- 当前需求是**新增原生客户端**而非简单修补
- 功能覆盖面广，包含实时聊天、媒体管理、长列表、工具模块、设置模块
- 需要一套能长期维护的状态管理与模块边界
- Compose 更适合复杂移动端界面及后续持续扩展

## 7. 成功标准

### 功能成功标准
- Android 端可完成登录、身份选择、会话列表、聊天、上传与历史媒体等主流程
- Android 端可对接抖音、mtPhoto、视频抽帧模块
- 认证、上传、消息历史、WebSocket 行为与当前网页端保持一致

### 技术成功标准
- 网络层可同时支持 form、json、multipart、流式下载、WebSocket
- 本地缓存与 UI 状态边界清晰，具备离线恢复与异常兜底能力
- 模块拆分合理，可支持后续逐步测试和扩展

### 体验成功标准
- 常用路径（登录、选身份、聊天、上传）可在单手操作下顺畅完成
- 大图、视频、长列表有基础性能保障
- 外部依赖不可用时，模块能清晰提示而不是整体崩溃

## 8. 风险与约束

### 已知约束
- 服务端接口风格不统一，不能强行套一个统一 BaseResponse
- WebSocket 必须使用 `?token=` 握手，并在连接后立即发送 `sign`；`sign` 不仅包含 `id`，还包含 `name/userSex/userip/useraddree/randomvipcode` 等字段
- WebSocket `forceout` 的禁止重连时长以代码为准，当前为 **5 分钟**（`internal/app/forceout.go`），不是 80 秒
- 聊天消息方向不能只依赖 `fromUserId == currentUserId` 判断，网页端实际按 `fromuser.id === md5(identity.id)` 推断是否为“自己发送”
- 聊天页还依赖 `reportReferrer`、`ShowUserLoginInfo`、`warningreport` 等主动动作，Android 端需要保持兼容
- 媒体地址可能依赖当前服务访问域名/端口，Android 需使用设备可达地址
- 抖音、mtPhoto、抽帧能力受服务端环境变量与外部服务可用性影响

### 主要风险
- 功能面过大，若一次性开发，集成风险高
- 聊天页与媒体页状态复杂，若边界不清晰容易导致消息/上传状态错乱
- 长视频/大文件上传、图片预览、历史分页会带来性能压力

### 风险应对方向
- 先建立清晰分层与模块边界
- 网络、存储、实时通信采用统一基础设施层
- 在开发前补齐 WebSocket 协议目录（code 表、主动动作、消息方向判定）
- 对外部能力做“可用性检测 + UI 降级”
- 以任务清单分模块推进，优先保证核心主链路质量

## 9. 假设与待确认项

⚠️ 不确定因素：以下内容当前未由用户明确指定，本提案基于默认假设推进。

- 假设 1：Android 端首选 **Kotlin** 而非 Java
- 假设 2：UI 采用 **Jetpack Compose** 而非 XML
- 假设 3：最低系统版本默认建议 **Android 8.0（API 26）**，如需更低版本可在实施前调整
- 假设 4：允许采用“同一架构下分批完成功能”的实施节奏，但最终目标仍是功能对齐网页端
- 假设 5：Android 端默认复刻网页端的 `reportReferrer` 上报流程，以及 `ShowUserLoginInfo` / `warningreport` 等主动 WebSocket 动作

如后续用户对最低版本、UI 风格、发布节奏有新要求，应回到需求分析/方案调整流程。
