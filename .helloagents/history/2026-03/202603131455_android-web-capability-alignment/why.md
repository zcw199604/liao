# Android / Web 能力对齐提案

## 1. 背景

当前仓库已经同时存在两套客户端实现：
- **Web 端**：`frontend/`，能力最完整，已经形成登录 → 身份 → 会话列表 → 聊天 → 收藏 / 媒体 / 系统设置的完整业务闭环。
- **Android 端**：`android-app/`，已落地原生客户端骨架、基础登录、身份管理、会话列表、聊天房间、WebSocket 连接与基础设置页。

最近一轮对比审查表明：
- Android 与 Web 在**主链路行为**上仍存在明显差异，尤其是登录失效处理、forceout、身份编辑同步、会话列表实时更新、消息回显去重、富媒体消息与系统入口能力。
- Android 已声明了部分 API / 协议 / 本地缓存结构，但尚未与 UI 和业务闭环完全接通。
- Web 端已经形成事实标准；如果 Android 不持续对齐，后续会出现“同一个产品、两套不同规则”的维护成本与用户困惑。

因此，本轮不再局限于单点修复，而是启动一次**跨端能力追平与行为对齐**工作：
- 以 **Web 现有行为** 作为主要对齐基准
- 将 Android 从“最小可用文本版”推进到“覆盖当前 Web 主体能力的原生客户端”
- 同时对两端的能力矩阵、协议理解和状态模型进行收敛，降低后续分叉风险

---

## 2. 目标

### 2.1 核心目标
- 以 Web 端当前实现为基准，对齐 Android 端的核心业务行为与能力入口。
- 消除主链路上的行为不一致，确保同一业务操作在两端的结果一致或接近一致。
- 将 Android 侧未接通的 API、WebSocket 事件、媒体能力、收藏能力、系统能力补齐到可用状态。
- 沉淀统一的“跨端能力矩阵”和“对齐原则”，避免后续继续分叉。

### 2.2 交付目标
- 登录 / 失效 / forceout / 退出登录链路对齐
- 身份创建 / 编辑 / 删除 / 选择 / 会话恢复链路对齐
- 会话列表实时更新、未读、收藏、全局收藏链路对齐
- 聊天房间的历史分页、发送回显、失败重试、WS 事件消费对齐
- 图片 / 视频 / 文件消息的最小可用收发与展示对齐
- 设置 / 系统设置 / 媒体中心 / mtPhoto / 抖音 / 视频抽帧入口与主流程尽量追平
- 更新知识库文档，明确 Android / Web 的统一能力边界

---

## 3. 范围

### 3.1 范围内

1. **认证与会话生命周期对齐**
   - Token 校验失败处理
   - forceout / reject 处理
   - 退出登录状态清理
   - 当前身份恢复与会话恢复

2. **身份管理对齐**
   - 身份列表、快速创建、创建 / 编辑 / 删除 / 选择
   - `updateIdentityId` 支持
   - 名字 / 性别修改后的在线状态同步（`chgname` / `modinfo`）

3. **会话列表对齐**
   - 历史 / 收藏列表
   - 列表项实时更新：最后消息、时间、未读数、置顶
   - 全局收藏真实页面与入口
   - 未读清零策略统一

4. **聊天房间与 WebSocket 对齐**
   - 历史消息分页 / 增量加载
   - optimistic send / echo merge / retry
   - 文本 / 图片 / 视频 / 文件消息支持
   - typing、在线状态、匹配、warningreport、randomOut、forceout 等 WS 事件
   - 房间头部收藏、拉黑、清空重载、聊天历史媒体等入口

5. **媒体与扩展能力对齐**
   - 上传菜单、已上传媒体、聊天历史媒体
   - 图片管理、全局媒体库、图片查重
   - mtPhoto 浏览与导入上传
   - 抖音解析 / 下载 / 导入上传 / 收藏管理
   - 视频抽帧任务创建、任务中心、结果消费

6. **设置与系统能力对齐**
   - 身份信息设置
   - 系统设置、连接统计、断开全部连接、forceout 清理
   - 图片端口策略、主题偏好等

7. **知识库与验收**
   - 更新模块文档、变更记录、对齐说明
   - 补充测试与对齐验收清单

### 3.2 范围外
- 重写 Go 后端接口风格
- 引入与现有后端不兼容的新协议
- iOS 客户端对齐
- 完全像素级复刻 Web 样式

## 4. 用户与场景分析

### 4.1 用户画像
- **聊天主流程用户**：主要依赖登录、身份切换、列表、聊天、收藏等高频操作。
- **媒体重度用户**：依赖图片上传、聊天历史图片、媒体管理、mtPhoto / 抖音导入。
- **高级功能用户**：使用视频抽帧、全局收藏、系统设置与连接管理能力。
- **维护者**：关注 forceout、连接状态、图片端口策略、全局能力一致性。

### 4.2 关键场景
1. 登录后自动恢复会话，身份与连接状态正确恢复。
2. 修改身份后，对端感知与本地状态保持一致。
3. 收到新消息时，会话列表能立即更新且未读正确。
4. 自己发送消息时，不出现重复消息或状态不明。
5. 在 Android 上也能完成 Web 已有的收藏、媒体、扩展工具主流程。
6. 被 forceout 或 token 失效时，两端表现一致、可理解、可恢复。

### 4.3 当前痛点
- Android 与 Web 行为不同，导致排查线上问题时缺乏统一标准。
- Android 已有的接口封装没有完全接上页面和状态层，形成“半成品能力”。
- Web 功能越来越多，而 Android 如果继续按页面补丁式推进，后续状态管理会失控。
- 媒体与扩展模块入口分散，若缺少统一能力矩阵，容易出现“功能有接口、没页面；有页面、没闭环”。

## 5. 方案对比

## 方案 A：仅修复 P1 主链路差异

### 优点
- 风险最低，交付最快
- 能先消除登录 / 聊天核心体验问题

### 缺点
- 无法满足“对齐两端能力”的完整目标
- 全局收藏、媒体、mtPhoto、抖音、抽帧等扩展能力仍然分叉

### 适用性结论
适合作为短线止血，但不适合作为当前目标的最终方案。

## 方案 B：全量能力一次追平（已选）

### 优点
- 直接围绕“跨端统一产品能力”推进
- 主链路与扩展能力一起收敛，避免后续重复返工
- 便于一次性建立 Android 侧稳定的状态模型与模块边界

### 缺点
- 变更面最大
- 需要同时处理状态、协议、UI、媒体、扩展功能与文档
- 交付周期与回归成本显著高于单点修复

### 适用性结论
适合当前用户明确要求“对齐两端能力”的目标，是最完整但风险最高的方案。

## 方案 C：先重构 Android 状态层，再分阶段补功能

### 优点
- 架构更稳，后续扩展更顺滑
- 能先统一 Android 内部状态模型

### 缺点
- 短期可见收益较低
- 用户感知到的功能差异消除速度较慢

### 适用性结论
可作为技术实施节奏，但不单独作为本轮交付方案。

---

## 6. 选型结论

本次采用：**方案 B：全量能力一次追平**。

选型理由：
- 用户已经明确选择“全量能力一次追平”而非仅修 P1 差异
- 现有 Android 侧已经具备一定基础设施，继续只修局部会造成更多中间态
- Web 端已经形成事实基准，现在是统一产品行为的最佳时机
- 通过一次完整对齐，可以同时建立 Android 侧长期可维护的状态模型与能力边界

---

## 7. 成功标准

### 7.1 行为成功标准
- 登录失效、forceout、退出登录、身份恢复在两端行为一致
- 身份编辑 / 删除 / 切换后的在线状态与会话状态一致
- 会话列表能实时体现消息、未读、收藏与排序变化
- 文本消息不重复、失败可感知、回显能正确合并
- Android 可消费至少与 Web 同等级别的核心 WS 事件
- Android 能打开并完成 Web 侧已有的主要媒体 / 收藏 / 系统能力入口闭环

### 7.2 工程成功标准
- Android 建立统一的会话 / 消息 / WS 事件 / 媒体状态层
- 富媒体消息模型在 Android 侧完成最小统一抽象
- 知识库更新为“跨端能力矩阵 + 对齐基准”
- 关键验证命令、核心回归清单和风险项明确记录

### 7.3 验收标准
- Android 关键模块编译通过，核心测试可运行
- `go test ./...` 通过
- Web 构建门禁（`frontend npm run build`）保持通过
- 对齐清单中的关键差异项关闭或降为已知限制
