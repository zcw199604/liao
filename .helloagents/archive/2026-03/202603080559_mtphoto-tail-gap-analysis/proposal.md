## 1. 需求

### 背景
在补完 `mtphoto` 高价值回退链路测试后，`frontend/src/stores/mtphoto.ts` 的覆盖率已提升到：
- statements: 99.52%
- branches: 98.98%
- functions: 100%
- lines: 100%

用户进一步要求继续处理剩余低收益/边缘分支，因此需要先判断这些分支是否仍然可达且值得继续补测。

### 目标
- 明确 `mtphoto.ts` 当前剩余未覆盖分支的位置与可达性。
- 只为“可达且有业务价值”的分支补测试；若剩余分支属于结构性不可达或极低收益，则停止继续补测并记录原因。
- 保持测试可维护性，不为追求 100% 写脆弱测试。

### 非目标（本轮不做）
- 不为了命中不可达分支去重构生产代码。
- 不通过 hack / 直接调用未导出内部 helper 的方式制造覆盖率。
- 不继续扩大测试范围到与 `mtphoto` 无关的模块。

### 约束条件
```yaml
修改范围: 若无必要，不新增代码与测试改动
判断依据: 以最新 coverage-final.json 与 mtphoto.ts 当前实现为准
原则: 优先业务价值与可维护性，不盲目追求 100%
```

### 验收标准
- [x] 明确剩余未覆盖点的行号与类型。
- [x] 对每个剩余点给出“可达 / 疑似不可达”的判断依据。
- [x] 若无值得继续补测的点，则明确记录“本轮不再新增测试”的结论。

## 2. 方案

### 技术方案
基于 `frontend/coverage/coverage-final.json` 与当前 `mtphoto.ts` 实现做静态分析：

1. 读取 `mtphoto.ts` 剩余未覆盖 statement / branch。
2. 逐个对照源码调用路径，判断是否能通过**公开 store API** 触达。
3. 若某分支仅能通过直接调用未导出 helper、或前置条件已在更上游被短路，则认定为“结构性不可达/不值得补测”。
4. 若全部剩余点均属此类，则本轮结束，不新增测试代码。

### 影响范围
```yaml
涉及模块:
  - mtphoto: 剩余 coverage 尾差分析
预计变更文件: 方案包本身（无业务代码与测试文件改动）
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 误把少量可达分支判断为不可达 | 中 | 以 coverage-final.json + 实际公开调用路径双重校验 |
| 为追求 100% 覆盖率引入脆弱测试 | 高 | 本轮明确拒绝不可维护的测试写法 |

## 3. 技术设计（可选）

### 结论摘要
当前剩余未覆盖点主要为：
- `line 125`：`firstCoverMD5()` 中 `first ?? ''` 的右侧分支
- `line 158`：`mapFolderNodes()` 的 `!Array.isArray(rawList)` 分支
- `line 201`：`mapMediaItems()` 的 `!Array.isArray(rawList)` 分支
- `line 935`：`segments.length === 0` 分支
- `line 949`：`String(node.name || '')` 中 `|| ''` 的右侧分支

这些点在当前实现下都缺少通过公开 API 触达的现实路径：
- `125`：`primary.split(',')[0]` 对非空字符串总会返回字符串，不会得到 `undefined`
- `158/201`：所有公开调用点在进入 helper 前已做 `Array.isArray(...) ? ... : []` 预处理
- `935`：`normalizedPath === '/'` 已在上一行提前 return，无法再进入 `segments.length === 0`
- `949`：`mapFolderNode()` 已把空名称标准化为 `目录 {id}`，回调中 `node.name` 不会为空
