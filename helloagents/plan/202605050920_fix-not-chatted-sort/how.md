# 技术设计: 修复未聊天筛选导致的用户列表排序异常

## 技术方案
### 核心技术
- Vue 3 + Pinia store
- TypeScript 数组合并与稳定排序
- Vitest 单元测试

### 实现要点
- 在 `frontend/src/stores/chat.ts` 中调整 `loadHistoryUsers()` 的历史 ID 合并逻辑。
- 将当前 `historyUserIds.value = [...matchedNotChattedIds, ...newHistoryIds]` 改为“先合并去重，再按预览时间稳定排序”。
- 排序时间优先级使用 `lastMessageTime`，其次 `lastTime`；无法解析时保留合并前相对顺序。
- 保留 `matchedNotChattedIds` 的存在意义：后端历史列表未返回的匹配成功用户仍进入合并列表。
- `ChatSidebar.vue` 的未聊天筛选逻辑原则上不需要改动，因为它只做过滤，不是排序根因。

## 架构设计
无架构变更。本次修复限定在前端状态管理层，保持 UI 筛选层职责不变。

## 架构决策 ADR
### ADR-20260505-01: 未聊天用户保留但不默认置顶
**上下文:** 新增未聊天筛选后，为避免匹配用户刷新丢失，将本地未聊天用户前置合并到历史列表，破坏了默认消息列表的最近会话排序。
**决策:** 保留本地未聊天用户，但通过统一的时间排序决定其位置；未聊天聚合展示交给筛选按钮完成。
**理由:** 默认消息列表应服务“最近会话”心智，未聊天用户的快速访问应由筛选功能提供，而不是改变默认排序。
**替代方案:** 将未聊天用户追加到末尾 → 拒绝原因: 当匹配时间确实较新时会被错误压低。
**影响:** 默认列表顺序恢复为时间驱动；未聊天用户仍不会刷新丢失。

## 安全与性能
- **安全:** 不涉及认证、权限、PII 写入或服务端数据修改；无 EHRB 风险。
- **性能:** 历史列表合并排序复杂度为 `O(n log n)`，用户列表规模通常较小；如需优化，可仅在 `loadHistoryUsers()` 成功后执行一次排序。
- **稳定性:** 使用稳定排序兜底索引，避免时间字段缺失导致列表随机跳动。
