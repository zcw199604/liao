# 媒体预览视频操作重构提案

## 背景

当前 `MediaPreview` 同时服务 mtPhoto 相册、全站媒体库、抖音下载、抽帧任务中心等入口。视频预览层内同时暴露下载、上传/导入、保存当前帧和抽帧任务入口，导致用户在“查看视频”和“处理视频”之间频繁切换心智。

现有文案中“抓帧”和“抽帧”语义接近，但实际行为差异很大：

- 抓帧是即时操作：暂停当前视频，使用 canvas 截取当前播放位置的一帧，下载 PNG，并在有身份时上传到图片库。
- 抽帧是任务操作：进入抽帧创建弹窗，按关键帧、FPS 或范围批量生成图片。

这两个操作并列显示在视频预览底部和全屏浮层中，容易让用户误判操作结果，也挤占了不同来源入口的主业务动作。

## 问题陈述

### 1. 预览层职责过重

`MediaPreview` 既是图片/视频查看器，又承担视频工具入口、媒体导入入口、同图查询入口、抖音作者跳转入口等职责。视频场景中，底部主操作区出现“抓帧 / 抽帧 / 上传”并列，主次不清。

### 2. 文案不区分即时动作和任务动作

“抓帧”和“抽帧”都包含“帧”，但一个立即产出单张图片，另一个创建批量任务。按钮并列时，用户需要理解内部实现才能预测点击后的结果。

### 3. 不同来源入口使用同一套工具暴露策略

- mtPhoto 相册的主要目标是浏览并导入媒体。
- 全站媒体库的主要目标是管理历史媒体、重新上传或发送前加载。
- 抖音下载的主要目标是解析、预览、下载和导入。
- 抽帧任务中心的主要目标才是批量抽帧。

这些入口当前复用统一预览组件，但缺少场景化动作配置，导致“创建抽帧任务”在部分入口显得突兀。

## 目标

1. 明确区分“保存当前帧”和“创建抽帧任务”。
2. 将预览层主按钮聚焦到当前入口的核心动作。
3. 将视频处理类低频动作收进稳定的“视频工具”入口。
4. 保持现有后端接口和抽帧能力不变，优先做前端交互和文案重构。
5. 降低改动风险，不引入新的播放器或抽帧实现。

## 非目标

- 不重写视频播放器核心逻辑。
- 不调整 ffmpeg/ffprobe 后端抽帧算法。
- 不改变媒体文件存储路径和数据库结构。
- 不改变抖音下载代理、mtPhoto 导入、全站媒体库分页接口。

## 用户场景

### 场景 A: mtPhoto 视频预览后导入

用户从 mtPhoto 相册点开一个视频，主要期望是确认内容并导入本地。预览层应优先展示导入动作，抽帧任务作为次级工具入口。

### 场景 B: 全站媒体库中查看历史视频

用户打开历史媒体库，主要期望是管理、重新上传或发送前加载。视频工具不应抢占管理和发送动作。

### 场景 C: 抖音视频预览

用户解析抖音作品后，主要期望是预览、下载或导入。若想从视频取图，应先理解这是“保存当前帧”或“创建抽帧任务”，而不是看到两个近似按钮。

### 场景 D: 批量抽帧

用户明确进入抽帧任务中心或从视频工具选择创建任务后，才进入参数表单，选择关键帧、FPS、范围和最大帧数。

## 成功标准

- 视频预览底部不再并列展示“抓帧 / 抽帧 / 上传”三个主按钮。
- 所有“抓帧”文案统一改为“保存当前帧”。
- 所有任务型“抽帧”入口统一表达为“创建抽帧任务”或“批量抽帧”。
- `VideoExtractCreateModal` 头部按钮从“预览/抓帧”改为“预览源视频”。
- mtPhoto、全站媒体库、抖音下载三类入口可以按场景控制预览层主按钮和视频工具显示。
- 前端构建通过，现有视频预览、导入、抽帧任务创建路径不回退。

## 风险

- 预览组件复用范围广，动作显示规则调整可能影响聊天历史预览、查重弹窗或抽帧任务中心的嵌套预览。
- 全屏浮层和底部工具栏已有复杂状态，移动按钮位置时需避免遮挡 Plyr 控件。
- 抖音视频可能来自 `/api/douyin/download`，可预览但不一定能直接抽帧；入口可用性必须继续依赖现有 `openCreateFromMedia` 门禁。
