# 变更提案: 匹配未聊天用户归档

## 需求背景
匹配成功但未聊天的用户只存在于当前前端内存和上游即时事件中。再次刷新历史列表时，上游不会返回这类用户，导致用户无法继续看到刚匹配过的人。

## 产品分析

### 目标用户与场景
- **用户群体:** 使用多个本地身份进行匿名匹配聊天的用户。
- **使用场景:** 用户匹配到新人后未立即聊天，稍后刷新列表或切换身份再查看列表。
- **核心痛点:** 未聊天匹配用户会从列表消失，且不同身份列表需要互相隔离。

### 价值主张与成功指标
- **价值主张:** 匹配成功即保留可见入口，即使没有消息历史也能再次找到该用户。
- **成功指标:** 刷新历史列表后仍能看到匹配未聊天用户；上游返回用户顺序保持不变；不同身份列表不串数据。

### 人文关怀
避免用户因刷新或身份切换丢失短暂匹配对象，降低焦虑和重复匹配成本；不新增敏感数据采集，仅复用上游已返回的用户公开快照。

## 变更内容
1. 后端 WebSocket 收到 `code=15` 匹配成功事件时，将匹配用户快照写入 `chat_user_archive` 的 history 来源。
2. 历史列表继续复用现有归档合并逻辑，仅在上游列表末尾补齐缺失归档用户，不改变上游返回顺序。
3. 前端 chat store 记录列表所属身份，身份切换时清空旧身份列表再加载新身份列表。

## 范围边界
- **范围内:** Web 端聊天列表、后端 WebSocket 匹配事件归档、现有本地归档合并链路。
- **范围外:** Android 客户端、数据库表结构新增、收藏列表语义变更、上游接口协议变更。
- **拆分说明:** 本需求可形成单个可验证闭环，不拆分。

## 影响范围
- **模块:** Chat UI、User History、WebSocket Proxy、Identity。
- **文件:** `internal/app/websocket_manager.go`、`frontend/src/stores/chat.ts`、相关测试与知识库。
- **API:** 无新增公开 API。
- **数据:** 复用 `chat_user_archive` 表，无迁移变更。

## 核心场景

### 需求: 匹配未聊天用户保存
**模块:** User History
匹配成功事件应生成本地归档快照，使未聊天用户在历史列表刷新后仍可见。

#### 场景: 上游刷新不返回未聊天匹配用户
- 上游历史列表原有顺序保持不变。
- 缺失的匹配用户从本地归档追加显示。

### 需求: 多身份列表隔离
**模块:** Chat UI
前端列表状态应绑定当前身份，切换身份后不复用旧身份列表。

#### 场景: 身份 A 切换到身份 B
- 身份 A 的历史/收藏列表不会在身份 B 页面残留。
- 身份 B 加载自己的后端列表和归档数据。

## 风险评估
- **风险:** WebSocket 消息处理路径增加数据库写入可能影响实时消息处理。
- **缓解:** 单条匹配事件只写一条归档记录，使用已有服务和最小字段快照，不新增依赖或迁移。
