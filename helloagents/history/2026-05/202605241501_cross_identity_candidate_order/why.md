# 变更提案: 修复跨身份联系人候选顺序

## 需求背景
跨身份联系人接入功能已经可以从其他身份的历史、收藏和本地归档中展示候选联系人，但当前候选列表的显示顺序与用户实际查看历史/收藏列表时的顺序不一致。

现有实现按“上游历史 → 上游收藏 → 本地归档”的固定合并顺序输出，并且本地归档按 `last_seen_at DESC, updated_at DESC, id DESC` 排序；同一批归档用户会共享相同时间戳，容易退化为数据库自增 ID 排序，导致用户难以按熟悉的列表位置找到联系人。

## 产品分析

### 目标用户与场景
- **用户群体:** 使用多个身份在聊天室中切换、并需要从其他身份接入已有联系人继续沟通的用户。
- **使用场景:** 用户打开“从其他身份接入”，选择来源身份后，希望列表顺序与该身份历史/收藏列表实际查看顺序一致。
- **核心痛点:** 联系人虽然存在，但顺序错乱，增加查找成本，并容易误判为“联系人缺失”。

### 价值主张与成功指标
- **价值主张:** 保持跨身份候选与原身份实际列表顺序一致，降低查找和确认成本。
- **成功指标:**
  - 上游历史返回顺序在候选列表中被保留。
  - 上游收藏返回顺序在候选列表中被保留。
  - 本地归档仅补充上游未返回的联系人，不打乱已有候选顺序。

### 人文关怀
保持用户熟悉的信息顺序，减少认知负担；不额外暴露跨身份以外的数据，不新增敏感信息采集。

## 变更内容
1. 调整跨身份候选聚合排序：保留来源列表原始顺序，避免数据库归档排序覆盖上游顺序。
2. 为本地归档增加稳定排序策略或位置字段规划，使纯归档场景尽可能贴近最近实际列表顺序。
3. 增加后端测试覆盖历史/收藏/归档混合合并时的顺序稳定性。
4. 更新知识库，明确跨身份候选的排序规则。

## 影响范围
- **模块:** 后端联系人候选接口、用户归档服务、前端跨身份联系人弹窗、知识库。
- **文件:**
  - `internal/app/user_history_handlers.go`
  - `internal/app/user_archive.go`
  - `internal/app/user_history_archive_handlers_test.go`
  - `frontend/src/components/chat/CrossIdentityContactPicker.vue`
  - `helloagents/wiki/modules/user-history.md`
  - `helloagents/wiki/modules/chat-ui.md`
  - `helloagents/CHANGELOG.md`
- **API:** `GET /api/chat/contactCandidates` 响应字段可保持兼容，必要时可新增可选排序元数据字段。
- **数据:** 优先不做破坏性迁移；如需持久化列表位置，新增可空字段并兼容旧数据。

## 核心场景

### 需求: 保留上游历史顺序
**模块:** User History / Chat UI
当来源身份的历史列表返回 `[A, B, C]` 时，跨身份候选中历史来源联系人应按 `[A, B, C]` 展示。

#### 场景: 历史列表原序输出
来源身份历史接口正常返回多个联系人。
- 候选列表中历史来源联系人顺序与上游返回顺序一致。
- 本地归档中同 ID 用户只补充标签和字段，不改变其位置。

### 需求: 保留上游收藏顺序
**模块:** User History / Chat UI
当来源身份收藏列表返回 `[F1, F2]`，且这些联系人不在历史列表中时，应按收藏返回顺序追加展示。

#### 场景: 收藏列表追加输出
历史列表为空或不包含收藏联系人，收藏接口正常返回多个联系人。
- 候选列表按收藏接口返回顺序追加收藏联系人。
- 候选项正确标记 `favorite` 来源。

### 需求: 归档只做补充且排序稳定
**模块:** User Archive
本地归档只补充上游当前未返回的候选，避免改变上游已确定的顺序。

#### 场景: 上游与归档混合
同一个用户同时存在于上游和归档。
- 上游出现位置优先。
- 归档只补充 `localArchived`、`history/favorite` 标签和缺失字段。
- 仅归档用户按稳定规则追加在末尾。

## 风险评估
- **风险:** 若新增数据库字段保存列表位置，可能涉及迁移兼容。
- **缓解:** 优先采用最小变更：在接口聚合层保留上游原始顺序；归档排序只作为补充。若确需持久化位置，则新增可空字段并为旧数据保留现有排序兜底。
