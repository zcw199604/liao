# 变更提案: 跨身份联系人接入

## 需求背景
当前聊天流程中，一个身份匹配到的用户主要停留在该身份的历史/收藏/归档上下文中。用户切换到另一个身份后，如果想直接联系前一个身份曾匹配到但未聊天的对象，需要重新匹配或手动查找，操作成本高。

项目已具备 `chat_user_archive` 本地归档能力，且后端在 WebSocket 收到 `code=15` 匹配成功事件时会保存匹配对象快照。因此可以基于现有归档数据和上游历史/收藏列表，为当前身份提供“从其他身份选择联系人并临时进入聊天”的入口。当前身份发送第一条消息后，上游会自然生成该身份与目标用户的聊天关系，后续历史列表可正常展示。

## 产品分析

### 目标用户与场景
- **用户群体:** 需要在多个本地身份之间切换并复用匹配资源的使用者。
- **使用场景:** A 身份匹配到 test 后未聊天；用户切换到 B 身份，希望用 B 身份直接进入与 test 的聊天页并发送消息。
- **核心痛点:** 不希望 B 身份重复匹配，也不希望只依赖上游当前返回列表而丢失本地已归档的匹配对象。

### 价值主张与成功指标
- **价值主张:** 降低跨身份复用联系人和发起会话的操作成本，同时保持上游关系由真实发送行为生成。
- **成功指标:** 用户能在 B 身份下通过按钮选择 A 身份的匹配/历史/收藏对象，进入聊天页并完成首条消息发送；刷新 B 历史列表后目标用户出现在 B 的历史联系人中。

### 人文关怀
该能力应避免误导用户“已经以 B 身份建立正式关系”。在首条消息发送前，目标用户应以临时接入状态展示，并在发送失败时保留清晰反馈，避免误操作和身份混淆。

## 变更内容
1. 增加跨身份联系人候选接口，合并来源身份的上游历史、上游收藏和本地 `chat_user_archive` 记录。
2. 在聊天列表/聊天页增加“从其他身份接入”按钮，支持选择来源身份和目标联系人。
3. 选中联系人后，以当前身份创建临时会话并跳转聊天页，不写入当前身份正式历史列表。
4. 当前身份首次发送消息成功或收到回显后刷新历史列表，使上游正式关系自然回流。
5. 调整前端消息缓存 key，按 `当前身份ID + 对方用户ID` 隔离，避免不同身份联系同一目标时串消息。

## 核心场景

### 需求: 从其他身份选择联系人
**模块:** Chat UI
当前已选择 B 身份时，用户可以点击入口，从 A 身份的联系人候选中选择 test。

#### 场景: 来源身份有本地归档匹配对象
给定 A 身份曾匹配 test，后端已将 test 写入 `chat_user_archive`
- 候选列表应展示 test，即使上游历史列表当前未返回 test。
- 选中 test 后，应停留在 B 身份并进入 `/chat/test`。

#### 场景: 来源身份有上游历史或收藏对象
给定 A 身份上游历史/收藏接口返回 test
- 候选列表应展示 test，并可标注来源为历史或收藏。
- 同一 targetUserId 在多个来源出现时只展示一条，优先保留更完整的快照。

### 需求: 当前身份临时接入会话
**模块:** Chat UI
选中 A 身份的 test 后，B 身份创建临时会话，但不立即污染 B 的正式历史列表。

#### 场景: 首条消息发送前
进入聊天页后应展示 test 的快照信息和空消息列表或 B 与 test 的历史。
- 发送 payload 的 `id` 必须是 B 身份 ID。
- 临时会话应有清晰状态，避免误认为已经写入 B 历史。

#### 场景: 首条消息发送成功
发送成功或收到自己消息回显后，前端刷新 B 的历史列表。
- 如果上游生成关系成功，test 应进入 B 的历史用户列表。
- 如果发送失败，test 仍保持临时会话状态，并展示发送失败/重试。

### 需求: 消息缓存按身份隔离
**模块:** Chat UI
A 和 B 都可能与同一 test 聊天，本地缓存必须区分两个会话。

#### 场景: A 与 B 分别联系同一目标
给定 A:test 和 B:test 都存在本地消息
- A 身份下看到的消息不应出现在 B 身份聊天页。
- B 身份发送的乐观消息、回显合并和加载更多不应影响 A:test。

### 需求: 候选数据包含数据库归档
**模块:** User History
候选接口需要包含上游当前返回和本地数据库存储的对象。

#### 场景: 上游不可用或目标已被上游列表删除
当上游历史/收藏接口失败或不再返回某个用户
- 接口仍应返回 `chat_user_archive` 中可用的候选。
- 候选项应标注 `localArchived=true` 或来源为 `archive`。

## 风险评估
- **风险:** 跨身份选择容易造成身份混淆。
  **缓解:** UI 明确展示“当前身份 B”和“来源身份 A”，发送仍强制使用当前身份。
- **风险:** 不隔离消息缓存会串会话。
  **缓解:** 将 Message Store 的 key 统一改为 `conversationKey(currentUserId, targetUserId)`。
- **风险:** 候选接口调用上游可能变慢。
  **缓解:** 支持 `includeUpstream=false` 或超时降级到本地归档；前端展示加载和失败状态。
- **风险:** 首条消息发送失败时目标不会进入 B 的上游历史。
  **缓解:** 保留临时会话与失败状态，提示重试，不把目标强行写入 B 的正式历史。
